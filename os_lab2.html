函数调用是软件层对程序执行的抽象，而其每一步操作（取指、访存）都通过 CPU 的虚拟地址访问机制完成，本质上函数调用过程就是一系列受 TLB 和页表管理的访存操作。

Thinking 2.7 从下述三个问题中任选其一回答：

  • 简单了解并叙述 X86 体系结构中的内存管理机制，比较 X86 和 MIPS 在内存管理上的区别。

  • 简单了解并叙述 RISC-V 中的内存管理机制，比较 RISC-V 与 MIPS 在内存管理上的区别。

  • 简单了解并叙述 LoongArch 中的内存管理机制，比较 LoongArch 与 MIPS 在内存管理上的区别。

RISC-V 的内存管理机制以标准化多级页表为核心，由 satp 统一描述当前地址空间，硬件按照既定的页表格式完成地址翻译，并通过 SFENCE.VMA 保证页表更新与翻译缓存的一致性；而 MIPS 更强调软件参与的 TLB 管理，TLB miss 后通常通过异常处理程序装入表项，操作系统需要更直接地操纵 TLB 和相关 CP0 寄存器。RISC-V 更偏向“硬件规定页表格式 + 统一翻译接口”，MIPS 更偏向“硬件提供 TLB 机制 + 软件负责 refill 与维护”。

这套链表宏是为一类特殊的双向链表设计的，这类链表大概长这样：

它和普通双向链表主要有两个区别：① 它的 prev 和 next 属性不是直接裸露在 Node 里的，而是用 LIST_ENTRY 属性封装好再放置于 Node 里的。这样的好处在于一个 Node 可以有不止一个 LIST_ENTRY，也就是可以有不止一组 prev 和 next，也就是这个 Node 可以同时被连进多个链表里（因为一段内存，它可以同时被记录为 free 和 dirty，所以表示内存的结构体需要具备同时处于多个链表中的能力）；② 它的 prev 不是指向前驱节点，而是指向前驱节点的对应 LIST_ENTRY 里的 next 。

现在可以知道 MOS 里页面的大小是 212B2^{12}B212B，而一个页目录摊开来的大小（页目录映射的最大范围，也就是一个进程享有的虚拟空间大小）是 232B2^{32}B232B。

va 的结构是：

PDX：页目录索引 PTX：页表索引 offset：页内偏移
10 10 12

这里 PDX 和 PTX 的作用就是把相应部分提出来挪到低位（PDX 也 & 0x3FF 了，其实不用吧）。

一个页表项 32 位，高 20 位是对应物理页框的地址的高 20 位（因为页表头要按 PAGE_SIZE 对齐，所以虚拟地址的低 12 位一定是 0），低 12 位是对应页表的各种权限。PTE_ADDR 和 PTE_FLAGS 分别是为了把这俩提出来。

PPN 和 VPN 是通过地址求页号。

cur_pgdir 始终指向当前生效的地址空间的页目录头部，切换进程的时候地址空间发生改变，cur_pgdir 的指向也会发生改变。它的类型是 Pde*，也就是 unsigned long*，这是在我们的代码中常见的地址指针类型。

然后才定义 Page 结构体（物理页面描述符），展开来看就是：

GPT 说这个东西叫做物理页面描述符。每个 Page 结构体用于描述一个页面，pp_link 描述当前页是否/是怎样被连到空闲页链表里的，pp_ref 描述当前页被几处引用（对于普通数据页，等价于有多少有效映射指向它；对于页表页、启动阶段占有页等具有特殊用途的页，也可能是手动置 1，比如 page_init 里）主要是为了反映能否把它 free 掉。显然 Page 并非物理页，但是鉴于物理页本身不好描述，用 Page 指代物理页好像也无可厚非。

注意我们不能真的把物理页面挂在链表里，所以实际上被挂在 Page_list 里的是其对应的页面描述符。

从 pmap.c 里 extern 来的 Pages 指向物理页面描述符的起点？

page_free_list 是一个 Page_list 类型（刚刚定义的）的链表，用来挂当前闲置页面的描述符。

函数 page2ppn 用于计算某一页（pp）的物理页号（这里应用了一点 C 语言指针知识）。

函数 page2pa 用于计算某一页（pp）的物理地址。

函数 pa2page 与 page2pa 正好反过来，用于寻找某一物理地址对应的页面，返回指向页面头部的指针。

函数 va2pa 的作用是根据给定的页目录 pgdir，查询虚拟地址 va 当前映射到哪个物理地址。

注意：任何一张页表（页目录或者页表）存储的都只能是物理地址，而 C 语言只能操作逻辑地址。我们用 C 语言操作逻辑地址去读页表，读到的却是物理地址，这或许是代码中 p->v/v->p 操作如此频繁的原因。
